# ADR 0007 — Toolchain: Swift + SwiftUI, native, via SwiftPM

- Status: Accepted
- Date: 2026-07-06
- Related: ADR 0001, ADR 0006

## Context

The implementation language was left open at project inception. Three candidates were
evaluated: native Swift + SwiftUI via SwiftPM, a Rust core behind a thin Swift shell,
and Tauri. The scaffolded Swift core (generated from `models/core.als` via `oxidtr`)
already exists in `src/Core/`; a concrete toolchain decision is needed before further
implementation work begins.

Key constraints:

- Metal / mlx-swift for inference (ADR 0001 path).
- `DISPATCH_SOURCE_TYPE_MEMORYPRESSURE` for pressure signals (ADR 0003/0004/0005).
- Apple Translation framework as a capability-gated fallback (ADR 0006).
- `MenuBarExtra` for the menu bar UI.
- Primary memory target: 8 GB unified memory.
- Model-first workflow: `models/core.als` is the single source of truth; `src/Core/`
  is generated by `oxidtr` and must not be edited by hand.

## Decision

**Native Swift + SwiftUI via SwiftPM.** No FFI, no WebView, no additional runtime.

### Repository layout

```
models/          # Alloy source of truth (models/core.als)
src/
  Core/          # GENERATED by oxidtr: Models.swift, Operations.swift,
                 #   Fixtures.swift, Tests.swift (excluded from build via Package.swift)
                 # Hand-written: Support.swift
  Lifecycle/     # hand-written: weight-lifecycle state machine
  Pressure/      # hand-written: pressure monitor and domain
  Time/          # hand-written: idle-timeout logic
  Residency/     # hand-written: residency floor and eviction policy
  Engine/        # hand-written: llama.cpp / Metal / mlx-swift glue
  Service/       # hand-written: translation service orchestrator
  CLI/           # hand-written: CLI entry-point helpers
mbt/             # console executable (swift run mbt)
tests/           # swift-testing suite
docs/            # architecture, ADRs (docs/decisions/), validation notes
docs/decisions/  # ADRs — settled engineering judgements (English-only)
git-hooks/       # pre-push hook (SwiftFormat + oxidtr-check)
```

### Platform minimum

**macOS 14.** mlx-swift requires it. The minimum rises to **macOS 15** when the
ADR 0006 Translation-framework fallback lands. `MenuBarExtra` (the menu bar UI) needs
only macOS 13, so the later UI work is unaffected by the current floor.

### Test framework

**swift-testing** (bundled with Xcode 16 / Swift 6). `src/Core/Tests.swift` (generated
by oxidtr) is excluded from the build target via `Package.swift`; it is consumed only
by the testing target.

### Allowed build-time dependencies

- **swift-argument-parser** — CLI argument handling for `mbt` (already minimal).
- **mlx-swift** — added in the engine PR for Metal-backed inference. It is not on the
  translation path in a network-call sense; the local-only invariant remains intact.

No new dependencies are added without an ADR or explicit justification.

### Model-first workflow

```bash
# Regenerate src/Core/ from the Alloy model:
oxidtr generate models/core.als --target swift --output src/Core

# Consistency gate (wired into git-hooks/pre-push):
oxidtr check --model models/core.als --impl src/Core
```

The `oxidtr check` step runs in the pre-push hook after SwiftFormat; a drift between
`models/core.als` and `src/Core/` fails the push.

## Consequences

- **FFI-free path** to Metal / mlx-swift / Translation framework and the M2
  `MenuBarExtra` UI. No bridging headers, no Rust build artifacts.
- **SwiftPM** is the single build system. No Xcode project file is required for
  CI/headless builds.
- `AGENTS.md` "Project Structure", "Development Setup", and "Build & Test" TBD
  sections are now filled (see that file).
- The pre-push hook gains a **SwiftFormat** block (guards on `Package.swift`) and an
  **oxidtr-check** block.
- macOS 14 floor is a hard constraint until the engine PR lands; bumping to 15 for
  the Translation fallback is a one-line `Package.swift` edit when ADR 0006 is
  implemented.

## Alternatives considered

- **Rust core + Swift shell**: rejected. The FFI boundary would cut through the
  residency logic (ADR 0003/0004), requiring every Apple framework call to cross the
  boundary anyway. Pure Swift has no bridging cost and gives direct access to all
  required Apple APIs.
- **Tauri**: rejected. A WebView + JS runtime adds memory overhead that conflicts with
  the 8 GB target. `DISPATCH_SOURCE_TYPE_MEMORYPRESSURE` and the Translation framework
  still require native bridging, so no platform complexity is actually removed.
